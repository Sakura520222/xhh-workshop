## 深度反思报告

### 1. 审查质量评估
- **覆盖度（优）**：精准捕捉到了“组件常驻挂载”这一架构变更带来的所有直接副作用。特别是识别出了 `svelte:window` 全局监听器在组件不销毁时的越界响应问题（Escape 键），以及 `display: none` 对 `position: fixed` 模态框的渲染树裁剪效果。
- **准确度（优）**：对 `scrollTo` 在隐藏元素上的 no-op 行为、短生命周期 `setTimeout` 不构成泄漏等技术细节的判断完全准确。明确将后端 `flush_messages` 并发缺陷排除在本次增量审查范围外，边界划分清晰，无误判。
- **完整性（良）**：充分理解了类 Vue 的 `keep-alive` 语义。微小遗漏在于：未提及 Agent 视图常驻后，随着用户长时间使用，聊天记录数组在内存中的累积潜力（虽然单页应用中通常可接受，但作为内存维度的完整性可补充）。

### 2. 发现的模式
- **代码模式**：采用原生 CSS `display: none` 替代框架级别的条件渲染（`{#if}`）来实现状态保持，这是一种轻量级且无额外抽象的“伪 keep-alive”模式。同时，采用了“视图守卫”模式来隔离全局副作用。
- **架构观察**：前后端存在明显的**状态管理复杂性不对称**。前端正在通过“常驻挂载”等手段加固状态，以抵御切换页面时的状态丢失；而后端（从历史上下文看）却深陷“全局活跃会话状态”的泥潭。前端的加固实际上是在补偿后端并发设计的脆弱性（如切换会话导致消息错乱）。

### 3. 规范完善建议
- **新增规则：【全局事件监听器生命周期守卫】**：当组件从“条件渲染（销毁/重建）”重构为“常驻隐藏（挂载/隐藏）”时，必须强制审查所有全局监听器（如 `svelte:window`、`window.addEventListener`），要求在回调内部增加当前视图/路由的活跃态校验。
- **新增规则：【CSS 隐藏与层叠上下文隔离】**：使用 `display: none` 或 `visibility: hidden` 模拟组件卸载时，必须验证子树内的 `position: fixed/sticky` 元素、`popover` 属性或 `Teleport/Portal` 节点不会泄漏到全局层叠上下文中干扰其他视图。
- **强化规则：【显式上下文传递】**（针对后端历史缺陷）：严禁在处理并发流式请求时依赖全局单例状态（如 `active_session`），必须强制要求在函数签名中显式传递 `session_id` 等隔离标识。

### 4. 经验教训
- **生命周期与可见性的解耦风险**：现代前端框架中，`onMount`/`onDestroy` 仅代表 DOM 挂载/卸载，不代表用户“可见”。审查时需建立**“生命周期 + 可见性”双轨思维**，特别关注 WebSocket 长连接、轮询定时器、全局键盘/鼠标事件在“挂载但不可见”状态下的行为。
- **副作用保留的双刃剑**：keep-alive 类模式保留了“好的状态”（如输入框内容），也会保留“坏的状态”（如异常的内部标志位、未关闭的弹窗）。未来审查需关注组件再次显示时，是否需要类似 `activated` 的钩子进行状态重置。

### 5. 增量审查反思
- **问题解决情况**：本次提交是一笔纯粹的前端架构优化，没有试图去修复后端的 `flush_messages` 缺陷，变更范围非常收敛，符合增量审查的预期。
- **新引入风险**：风险极低。唯一潜在风险是，如果未来 Agent 组件内部引入了更重的资源（如 WebGL 上下文、沉重的 Web Worker），常驻挂载可能导致后台性能损耗，但目前仅限于文本流，无影响。
- **覆盖度下降的盲区（核心反思）**：虽然作为增量审查，本次评分 9/10 且 approve 是合理的，但**存在系统性风险穿透的盲区**。审查摘要中虽然清醒地指出了后端 `flush_messages` 的严重并发缺陷，但决策仍是 `approve`。在增量审查中，如果已知代码库中存在高严重级的既有缺陷（且该缺陷直接影响本次新增特性的最终用户体验：前端保住了状态，但后端可能把消息发错会话），应当在 Review 决策中附加 **Blocker 跟踪项**或降级为 `Request Changes`（要求在 PR 描述中承诺修复计划），而非静默放行。增量审查不应成为对全局已知致命缺陷“视而不见”的借口。

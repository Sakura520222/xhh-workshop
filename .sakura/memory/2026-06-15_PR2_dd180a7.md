### 2. 发现的模式

*   **代码模式**：
    1.  **时间戳作为 ID**：`format!("t_{:x}", now)` 是一种快速但脆弱的 ID 生成模式，在单进程、低并发下可行，但在涉及持久化、多会话管理的场景下，其碰撞风险不容忽视。这提示项目可能缺乏统一的、健壮的 ID 生成策略。
    2.  **异步循环中的同步锁**：这是从传统同步代码向异步运行时迁移时常见的**反模式**。它表明开发者可能尚未完全适应异步编程中“锁持有时间最小化”的核心原则。

*   **架构观察**：
    1.  **混合状态管理**：`state.rs` 中定义了结构化的会话状态（`AgentSessions`），而 `commands.rs` 中的命令函数直接对其进行加锁操作。这暗示项目的“状态”与“命令处理”之间是**紧耦合**的，所有并发控制逻辑分散在各个命令中，难以统一维护和优化。
    2.  **前后端功能对齐**：前端新增了复杂的交互组件（`AgentSidebar`, `TemplateEditor`），后端对应了大量新命令。这体现了典型的**功能驱动型**开发，但缺乏对 API 边界、错误码、操作幂等性的系统性设计。

### 3. 规范完善建议

基于本次审查，建议新增或强化以下审查规范：

*   **【新增】并发安全审查要点**：
    1.  **锁持有时间检查**：严禁在持有 `Mutex`/`RwLock` 的情况下执行 I/O（文件、网络）、等待用户输入或执行其他耗时操作。锁的临界区应尽可能短小。
    2.  **锁与异步上下文**：审查使用 `tokio::sync::Mutex` 等异步锁的场景时，需额外关注 `loop` 或递归调用中的锁持有情况，评估其对运行时整体吞吐的影响。
    3.  **锁释放路径**：检查所有可能提前退出（`return`, `?`, `continue`）的分支，确保 `MutexGuard` 能被正确 drop。对复杂逻辑，建议使用块作用域 `{ let _guard = lock.lock().await; ... }` 来精确控制生命周期。

*   **【强化】唯一性标识生成规范**：
    1.  对于需要持久化或跨会话使用的 ID（如模板 ID、会话 ID），禁止使用易碰撞的 `时间戳` 或 `随机数`。
    2.  推荐使用 `UUID v4`、`nanoid` 或基于 `原子计数器 + 时间戳` 的复合策略。

*   **【新增】状态初始化与生命周期审查**：
    1.  对于修改全局或共享状态（如会话、模板）的函数，需检查其调用链上游是否已确保运行时环境就绪（如 `ensure_agent_runtime`）。
    2.  审查“删除”操作（如删除内置模板）时，需评估其可逆性以及对系统默认状态的影响。

### 4. 经验教训

*   **关注“锁”的作用域，尤其是异步代码**：本次审查最大的收获是，异步运行时中的同步锁问题比想象中更隐蔽、更危险。未来的审查中，凡涉及共享状态操作，必须将其视为审查重点，并追问：“这个锁会锁住什么？锁多久？有没有可能锁住整个事件循环？”
*   **从“功能实现”到“健壮性设计”的思维转变**：开发者容易首先关注功能是否实现。审查者的价值在于推动思考：**当功能处于异常路径（如高并发、操作中断、状态不一致）时，系统是否依然可靠？** 本次审查中的两个 Major 问题正是从这个角度发现的。
*   **前置审查 ID 生成策略**：任何涉及数据存储的新增实体（模板、会话、文档等），都应在设计阶段就审查其主键生成策略，避免后期出现数据混乱的代价高昂的修复。
